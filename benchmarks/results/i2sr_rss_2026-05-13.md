# I2_SR RSS Probe, 2026-05-13

Hardware: Intel Xeon Silver 4116, portable AVX2 llama.cpp build, no BLAS.
The runtime was built with `patches/llama-i2sr-row-scale-qtype.patch` applied
for this measurement, then the patch was reversed and the default source tree
was rebuilt.

Command:

```bash
python benchmarks/run_gguf_memory_probe.py \
  --models-json benchmarks/gguf_qwen15b_i2sr_x86act_manifest.json \
  --out-dir benchmark_results/gguf-rss-qwen15b-i2sr-x86act-2026-05-13 \
  --llama-bin-dir build-portable-avx2/bin \
  --threads 12 \
  --ctx-size 512 \
  --tokens 1
```

The probe runs `llama-cli` under `/usr/bin/time -v`, with `-c 512`, 12 threads,
CPU-only execution, and one generated token so the model is loaded and used.
The reported metric is peak resident set size from `/usr/bin/time`.

| artifact | file MiB | max RSS GiB | return code |
| --- | ---: | ---: | ---: |
| fixed row-scale `I2_SR` x86 ACT candidate | 1,211.3 | 1.250 | 0 |

## Interpretation

The fixed direct `I2_SR` artifact has the same measured `ctx=512` RSS class as
the row-scale `I2_S` prototype (`1.250 GiB`) and row-scale `TQ2_0`
(`1.257 GiB`). This supports the memory-side claim that the separate-qtype
candidate preserves the row-scale ternary memory advantage over FP16 while
remaining larger than the dense `Q4_K_M` baseline.
